# ADR-0002: One D1 database for everything

**Status**: accepted · **Date**: 2026-07-14 (documenting choices made 2026-07-06 → 07-13)

## Context

The app stores visitor logs, articles **and their images (BLOBs)**, custom pages, menu,
settings, members, sessions, relay/vpn channel configs, playground conversations, and — since
v1.0.0 — request metering, error logs and audit trails. Cloudflare offers KV, R2, Durable
Objects and Queues, each a "more correct" home for some of that data.

## Decision

Everything lives in **one D1 database** (`ipua-logs`). Images are D1 BLOBs (≤1.8 MB after
client-side compression) served through an immutable-cached `/img/{id}` route. Sessions are
D1 rows, not KV. Metering is D1 rows with indexes, not Analytics Engine.

## Consequences

**Won**: one backup, one console, one local emulator, one mental model; joins across
domains are trivial (`/api/admin/users` joins quota columns with `req_log` counts in one
query); tests get the entire persistence layer from a single miniflare binding; free-tier
limits (5 GB, 5M reads/day) are far beyond a personal site's needs.

**Paid**: image BLOBs make the database the largest object in any export and cap image size;
D1 is single-region (all writes cross the ocean for far-away visitors — masked by
`waitUntil` for logging paths); no pub/sub semantics, so "cleanup" is done opportunistically
(1% of requests purge old `req_log` rows); global rate-limit counters are approximate under
concurrency (COUNT-then-insert, no transactions across requests — acceptable at this scale).
*Update 2026-07-17: the approximate rate-limit counting is superseded by the Durable Object
limiter (ADR-0007); the D1 COUNT path remains only as its fallback tier.*

**Revisit when**: image storage approaches ~1 GB (move media to R2 — already in DEBT.md),
or any table needs >100 writes/sec sustained.
