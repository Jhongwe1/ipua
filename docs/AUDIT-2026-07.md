# Security audit, July 2026 — two rounds, and the miss rate of the first

> Scope: the whole application (`src/`, `test/`, `migrations/`, docs). Two independent
> passes, 2026-07-19 and 2026-07-21, by the same reviewer using **deliberately different
> methods**. Fixes shipped in v2.1.0 (2026-07-22).
>
> The headline result of this audit is not the list of defects. It is that **the first
> round missed six things, one of them an open redirect on a line 158 lines away from
> code it had already read and signed off**. That number is the most useful thing in this
> document, so it goes at the top rather than in a footnote.

## Why this document leads with its own failure

The first draft of this report was going to say: *"I found 3 real defects and correctly
refuted 4 false positives."* That framing is a trap. It claims the review was thorough,
which invites the reader to review the review — and a second pass found six more issues in
twenty minutes. A reader who finds a seventh has caught the document lying about itself.

Reporting the miss rate instead cannot be falsified in that way, because the weakness is
already conceded. It is also the more useful engineering artifact: a review process with a
known, measured failure mode can be improved, and this one was.

| | Round 1 (07-19) | Round 2 (07-21) |
|---|---|---|
| Method | file-by-file close reading | **category sweeps across the whole tree** |
| Real defects found | 3 | 6 (all missed by round 1) |
| False positives correctly refuted | 4 | 0 |
| Notable miss | — | `safeNext` open redirect, in `auth.ts` — a file round 1 read in full |

## What changed between the rounds, and why it worked

Round 1 read files. That produces a specific blind spot: **finishing a file feels like
finishing its risks**, so `auth.ts` was marked "reviewed" after its session and admin logic
checked out — and `safeNext`, 158 lines away in the same file, was read as prose rather
than as a redirect sink.

Round 2 stopped reading files and started tracing *categories* across the whole tree:

- every value that reaches a `Location` header → found 1.1
- every decode/parse that can throw outside a `try` → found 1.2
- every `SELECT *` whose result reaches a response body → found 1.3
- every anonymous path that writes to D1 → found 1.9b
- every authorization decision reading a request-controlled input → found 1.9a
- every request-body key that both parse paths interpret differently → found 1.8d

Five of the six came from asking "where else does this *kind* of thing happen?" — a
question that only works if you are not organizing the search by file. This is the durable
lesson, more than any individual bug.

## Round 1 — findings

### 1.4 `PUT /api/admin/settings` wrote as it validated 🟠

`settings.ts:157-272` validated and immediately wrote each of 14 keys in sequence. A
request whose *later* key failed validation returned 400 with the *earlier* keys already
committed to D1 — and the front end treats 400 as whole-request failure and does not
refetch. The UI and the database disagreed, and neither side knew.

**Fixed**: validate everything, collect statements, one `env.DB.batch()` (a single D1
transaction). A 400 now means nothing happened. Side benefit: ~14 round-trips → 1.
**Verdict**: defect. **Tests**: `settings.test.ts` — "全有全無" block, 4 cases including
"delete-key in the same batch" and a positive case proving the normal path still works.

### 1.5 Upstream error bodies could carry third-party keys into `errlog` 🟠

`chat.ts` puts the upstream's raw response into `detail` → `errlog`. Cheap OpenAI-compatible
resellers — the `custom` channel kind this project explicitly targets — routinely echo the
full API key back in error text. That table is then read by `/api/admin/errors`, backed up
to R2 daily, **and scanned every 5 minutes by `tgAlertScan`, which interpolates `msg`
straight into a Telegram message.** One leak, four destinations.

**Fixed**: `redactSecrets()` in `lib/observe.ts`, applied in `reportErrorNow` *before*
truncation (truncating first would leave a severed key that no longer matches its pattern).
Covers OpenAI/Anthropic `sk-…`, Google `AIza…`, this site's own `uak-…`, and Telegram bot
tokens. Applied to `audit()` summaries too, as defense in depth.

**Deliberately at the observe layer, not the call sites.** There are 12 call sites today and
there will be more; the property "secrets never reach errlog" should not depend on each
future author remembering. **Verdict**: defect. **Tests**: `observability.test.ts` — 5
credential shapes, plus "clean messages pass through byte-identical" and "the `Error.stack`
path is redacted too".

### 1.6 `visits.path` was not length-capped 🟡

`_middleware.ts:47`. Its neighbours (`ua`, `lang`, `referer`) are all clamped; `path` was
not, and Cloudflare accepts URLs up to ~16 KB. Anyone could write a 16 KB row to `visits`
using nothing but a long URL. **Fixed**: `.slice(0, 500)`, matching `referer`.
**Verdict**: defect (minor). **Tests**: `middleware.test.ts` — truncation, normal-path
passthrough, and a regression test pinning the other three limits.

## Round 1 — refuted (4)

Recorded because *not* filing these was also a decision:

1. **`goodOrigin` allows localhost on production** — real, but intentional and pinned by
   `test/unit/auth.test.ts:88-91`. Promoted to a written accepted risk with a reversal
   condition rather than filed as a bug (THREAT-MODEL §4.1).
2. **Front-end JS is outside the type-checking toolchain** — already disclosed in
   ADR-0008:29. A known, documented gap is not a finding.
3. **"Four front-end bugs"** — verified line by line; the code is correct in all four.
   They are type-inference complaints, and "fixing" them would change zero behaviour.
4. **`getCookie` looked like a cross-site DoS** — downgraded to self-inflicted only, after
   checking that `cc.cd` is on the Public Suffix List, so a sibling subdomain cannot set
   `Domain=.cc.cd`. **This verification is the single highest-value action in round 1**:
   without it, 1.2 below would have been filed at the wrong severity, and the fix might
   have been designed against the wrong threat.

## Round 2 — findings (all missed by round 1)

### 1.1 `safeNext` open redirect via backslash 🔴

```ts
return /^\/(?!\/)/.test(s) ? s.slice(0, 300) : "/";   // blocks //evil.com, not /\evil.com
```

Per the WHATWG URL spec, `\` is equivalent to `/` in a relative URL under a special scheme,
so browsers resolve `Location: /\evil.com` to `https://evil.com`. The guard blocked exactly
one of the two spellings.

Reachable end to end: `/auth/login?next=/\evil.com` → stored in the `ipua_oauth` cookie →
`callback.ts:59` → the `Location` header at `callback.ts:130`. `login.ts:73` is the same
sink via the other path.

**Fixed**: `/^\/(?![/\\])/`. **Verdict**: defect — the most serious of the audit.
**Tests**: unit cases for four hostile spellings *plus* an end-to-end test in
`oauth.test.ts` asserting both the value stored in the cookie and the `Location` finally
emitted, with the real assertion being `new URL(loc, ORIGIN).origin === ORIGIN` — "stays on
this site" is the property that matters, not "equals `/`".

**This is the miss that reframed the audit.** Round 1 read this file, in full, and moved on.

### 1.2 `getCookie` could 500 every SSR page 🔴

`decodeURIComponent("%")` throws, and `getSessionUser`'s call site (`auth.ts:146`) sits
**outside** the surrounding try/catch. `Cookie: ipua_sess=%` → 500 on all 11 SSR pages.

Severity is limited to self-harm: `cc.cd` is on the Public Suffix List, so no sibling domain
can plant a cookie on `uaip.cc.cd` (see refutation 4 — the same check that saved round 1
from over-reporting also correctly *down*graded this one).

**Fixed**: decode inside try/catch, falling back to the raw value. Chosen over deleting the
decode entirely so that a future cookie needing encoding still works, and over returning
`""` so a malformed value degrades to "invalid session" rather than "no session" — the
session id has a `^[a-z2-7]{20,64}$` check downstream either way.
**Verdict**: defect. **Tests**: three malformed-encoding shapes assert no throw.

### 1.3 `maskRow` had zero tests — the README's headline security claim was unguarded 🟠

Both READMEs state "upstream keys never leave the server". `maskRow()`
(`admin/relay/channels/index.ts:132`, with a near-duplicate in `vpn/channels/index.ts:51`)
consumes `SELECT *`. **Not one of the 384 tests asserted that a channel key is absent from
an admin response.**

`relay.test.ts:155` tested that the key is correctly injected *upstream*. Nothing tested
that it does not flow *back*. The direction was inverted — and with `SELECT *`, adding any
column later leaks it silently, with no test turning red.

> Round 1 filed this under "explicitly not doing: 321 tests already far exceeds the bar."
> **That argument form is wrong.** Adding tests is not a question of count, it is a question
> of *risk position*. The best-tested code here was the code that naturally attracts tests
> (relay forwarding, the DO, sanitization); the untested code was admin CRUD — which is
> where every line of secret handling lives. The distribution was exactly backwards.

**Fixed**: `test/int/admin-secrets.test.ts`, 6 cases across every admin endpoint that reads
a table containing a secret — relay channels list + update, VPN channels list + update,
admin settings (Telegram token), and the member-facing channel list.

Two deliberate design choices in that file:
- **Assertions run against the serialized response body**, not field-by-field. Field
  comparison would miss precisely the "someone added a column" risk that motivates it.
- **Every case also asserts the endpoint really worked** (row present, `has_key: true`,
  hint non-empty). Otherwise a 500 or an empty array would satisfy "no secret present" and
  the test would be decoration.

Verified non-vacuous by mutation: adding `leaked: r.api_key` to `maskRow` turns two cases
red; reverting turns them green.

### 1.7 The documentation was lying 🟠

**The only class of defect a reader can catch without opening the source**, which is why it
outranks every pure code issue here. One repository contained **three contradicting test
counts**: README said 321, the upgrade plan said 369, the suite actually ran 384. (369 was
itself an artifact — `grep` counts `it(` as 369 because `test/unit/fastsse.test.ts`
contains control characters and is skipped as binary.)

`docs/COMPARISON.md` was worse: it still described "Pages Functions + `wrangler pages
deploy`" six days after the migration to a single Worker (ADR-0006), claimed tokens are
"never priced" when Phase J had shipped pricing, and said "6k-line codebase" against an
actual 10,426 lines in `src/`.

**Fixed**: every number corrected, and — more importantly — `tools/check-docs.mjs` now runs
in CI beside the existing CSP and OpenAPI drift checks, asserting that every test-count
claim equals what vitest actually ran, that the E2E-flow count matches, that
`package.json` / `lib/site.ts` / README agree on the version, and that the comparison
table's architecture row still says Worker.

**The first version of this fix was itself wrong, which is worth recording.** It counted
occurrences of `it(` in the test sources and asserted from inside vitest. That number is a
*static approximation of a dynamic fact*: an `it()` written inside a `for` loop appears once
in source and runs N times. The README said 420, the suite ran 425, and the drift check
**passed** — a guard against wrong numbers, reporting a wrong number, in green. It was
caught only because the suite total was visible next to it in the same terminal output.

Rewritten to read `numTotalTests` from the run's JSON report, which is the fact itself
rather than a proxy for it. The general lesson is the one this whole audit keeps landing
on: *a check that can be satisfied without the property holding is not a check.* It is the
same failure as `expect([200, 401])` below, and the same failure as asserting "no secret in
this field" instead of "no secret in this response".

**Verdict**: defect (documentation). The interesting part is the fix class — *my docs fail
CI now*, so this cannot silently recur.

### 1.8d `n` was missing from `PROTECTED_BODY_KEYS` — silent content corruption 🟡

`playground.ts:139`. `fastsse.ts:23`'s `FIELD_RE` is a global regex that concatenates
**every** `"content":"…"` in a payload; the slow path reads only
`choices[0].delta.content` (`playground.ts:274`). With `n: 2`, an upstream returns two
choices per chunk — so the two parse paths produce **different text**.

Unreachable today (the playground never sets `n`), but `extra_body` accepts arbitrary keys
and `n` was not on the protected list, so an admin writing `{"n":2}` on a channel would
have triggered it. The failure mode is not a crash — it is **silent content divergence
between two code paths**, which is the hardest kind to diagnose. **Fixed**: one array
entry. **Verdict**: latent defect.

### 1.9a `adminOk` fail-open on missing config, keyed on a client-controlled header

```ts
if (env.LOGS_TOKEN) { if (await tokenEqual(token, env.LOGS_TOKEN)) return true; }
else if (isLocal(url)) return true;      // LOGS_TOKEN unset → decided by Host alone
```

`isLocal` reads `url.hostname`, which in Workers comes from the **Host header — supplied by
the client**. Not currently reachable: production sets the secret, and Cloudflare routes by
Host/SNI. But the shape is wrong twice over: an authorization decision taking a
request-controlled input, and a **fail-open on configuration error** — forget to set
`LOGS_TOKEN` and the gate silently degrades from "prove you have a key" to "claim the right
Host".

**Fixed by removing the category, not the instance.** `isLocal(url)` is gone; `isDevEnv(env)`
reads an explicit `DEV_UNSAFE_ADMIN` flag that lives only in `.dev.vars` — read by
`wrangler dev`, never uploaded by `wrangler deploy`, and deliberately *not* declared in
`wrangler.toml`. Production reads `undefined` → closed. All four former `isLocal` gates
(admin auth, `pgUser`, and both halves of the dev login form) now use it.
**Verdict**: wrong direction, not a live vulnerability. **Tests**: `adminok.test.ts` now
asserts a `Host: localhost` request is refused without the flag, that only the exact string
`"1"` opens it, and that the flag never overrides a wrong `LOGS_TOKEN`.

### 1.9b `/api/csp-report` — anonymous unlimited D1 writes, wired to the alert channel

`src/routes/api/csp-report.ts` had 10% sampling, length caps, and always returned 204 — but
was **anonymous and unrated**, inserting into `errlog`. That is the table `tgAlertScan`
polls every 5 minutes, and `cron.ts:50` interpolates `msg` into the message body.

Sustained POSTs therefore meant D1 write amplification **plus attacker-controlled text
arriving in the administrator's Telegram**.

**Fixed**: per-IP limiting through the existing `RATE_LIMITER` DO (a `csp` service
namespace, so it cannot interact with member quotas). Sampling runs *first* so 90% of
requests cost nothing — a DO call is cheaper than a D1 write but not free. Failure is
**fail-closed** (no limiter → no write), matching the demo-mode policy from ADR-0009: an
anonymous write path that cannot be bounded should close, and CSP reports are diagnostics
whose loss breaks nothing. **Verdict**: wrong direction. **Tests**: burst from one IP caps
at the limit, two IPs count separately, unbound limiter writes nothing.

## Not vulnerabilities — but pointing the wrong way

§1.9a and §1.9b share a property worth separating out: **neither is currently reachable,
and both fail toward "open."**

"Can't be reached today" and "is designed correctly" are different claims, and conflating
them is how a config change or a new route turns a dormant shape into a live vulnerability
without anyone re-reviewing it. `adminOk` failed open on missing configuration;
`csp-report` let anonymous text reach an administrator's alert channel. Neither had been
exploited. Both were fixed anyway, because the argument for leaving them ("nobody can get
there right now") is an argument about the current deployment, not about the code.

## The test suite audited itself

Two findings about the tests rather than the code:

- **`router.test.ts:113-116`** was named "admin API with key → 200" and asserted
  `expect([200, 401]).toContain(r.status)` — **it passed whether authorization succeeded or
  failed**. It was also the only place `api/admin/users/index.ts` was exercised under
  vitest, so that endpoint's authorization seam effectively had no net at all. Now asserts
  200-with-a-real-payload for the correct token and 401 for a wrong or missing one.
- **E2E gap, disclosed not fixed**: the 5 Playwright flows cover publishing, cross-identity
  streaming, VPN invisibility, demo quota exhaustion and `/api-docs` — **the flagship
  `/relay` path has no E2E coverage at all.** It is covered at the integration level
  (`relay.test.ts`, `relay-metering.test.ts`), so this is a gap in end-to-end confidence,
  not in correctness evidence. Listed here rather than quietly closed.
