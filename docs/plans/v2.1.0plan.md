# uaip v2.1.0 — 重擬版

> 2026-07-21。取代 2026-07-19 初版（原文保留在 git 歷史，`git show HEAD~1:v2.1.0plan.md`）。
>
> **重擬的兩個理由：**
> 1. 獨立第二輪稽核找到初版漏掉的 **6 個問題**，其中一個是開放轉址，就在初版當時正在讀的同一個檔案裡。初版的旗艦產出（「我做了一次完美的稽核」）因此從資產變成風險。
> 2. 站長自評「對這份 repo 的理解 50/50」。**這是比任何 code bug 都大的缺口**，計畫重心必須整個換掉。

## 這份計畫服務的對象（決定了下面每一個取捨）

| | |
|---|---|
| 身分 | 碩士生，投第一份正職 |
| 作品集 | **這是唯一的一個** |
| 時間 | 3 個月以上 |
| 自評 | 對 repo 的理解 **50/50**（部分大架構沒問題，部分沒消化） |

**推論：**

- 新鮮人的作品集門檻是課程專案／CRUD demo。這個 repo 客觀上在**前 5%**，起手牌很好。
- 正因為好，**「講不出來」的傷害被放大**。面試官對一個令人印象深刻但作者答不出來的 repo，會把「厲害」重新歸類成「不誠實」——那個印象不可逆。
- 「14 天、87 commit、15,000 行」對學生來說是更大的異常值。全部的重量都壓在現場對話上。
- 規模小（10 次 LLM 請求）與零協作訊號，**對新鮮人的扣分遠低於對資深職位**。不用為此焦慮。
- 3 個月很夠。不需要壓縮，但要按順序做。

**一句話版本：這個專案不缺程式碼，也不缺文件。它缺的是「作者完全擁有它」這件事，而那要靠做，不能靠讀。**

# 第 1 部分：地雷清除（約 3 小時）

**這一段不是「加分」，是「移除會扣分的東西」。**
現在 repo 裡有面試官 20 分鐘內找得到的缺陷、以及跟 `npm test` 跑出來對不上的自述數字。

每項先寫一個**修補前會紅**的測試，再修。

## 1.1 `safeNext` 反斜線開放轉址 —— `src/lib/auth.ts:79` 🔴

```ts
return /^\/(?!\/)/.test(s) ? s.slice(0, 300) : "/";   // 擋 //evil.com，沒擋 /\evil.com
```

依 WHATWG URL，special scheme 相對網址裡 `\` 等同 `/`，瀏覽器把 `Location: /\evil.com` 解析成 `https://evil.com`。

可達路徑：`/auth/login?next=/\evil.com` → `ipua_oauth` cookie → `callback.ts:59` → **`callback.ts:130` 的 `Location`**。`login.ts:73` 同樣。

**修**：`/^\/(?![/\\])/`。**測**：`test/unit/auth.test.ts` 已有 `//` 的「重點案例」，在旁邊補 `/\` 與 `/\/`。

## 1.2 `getCookie` 讓整站 SSR 500 —— `src/lib/auth.ts:73` 🔴

`decodeURIComponent("%")` 會 throw，而 `getSessionUser` 的呼叫點（`auth.ts:146`）在 try/catch **外面**。送 `Cookie: ipua_sess=%` → 11 個 SSR 頁面全部 500。

（僅能自傷——`cc.cd` 在 Public Suffix List 上，兄弟網域設不了 `Domain=.cc.cd`。**這是初版第 8 項的 PSL 查證救了自己一次，值得寫進稽核文件。**）

**修**：三個 cookie 值都是 base32 或 `1`，根本不需要解碼——直接拿掉 `decodeURIComponent`，或包 try/catch 回 `""`。

## 1.3 `maskRow` 零測試 —— README 頭號安全宣稱沒人守 🟠

`api/admin/relay/channels/index.ts:132` 的 `maskRow()` 吃 `SELECT *`（`:154`），`vpn/channels/index.ts:51` 還有一份複製品。**369 條測試裡沒有一條斷言渠道金鑰不出現在管理員回應中。**

`test/int/relay.test.ts:155` 測了金鑰正確**注入上游**，沒測它不會**流回來**。`SELECT *` 代表以後加欄位就靜默外洩。兩份 README 都寫著「Upstream keys never leave the server」。

**修**：兩條測試，各約 10 行——`GET /api/admin/relay/channels` 與 `/api/admin/vpn/channels`，斷言回應 JSON 序列化後**不含** `helpers.ts:162` 種下的 `sk-upstream-secret`。

> 初版把這歸進「明確不做：321 條已遠超標準」。**那個論證形式是錯的**——補測試不是看數量，是看風險位置。目前測得最徹底的是本來就會被測的地方（relay 轉發、DO、sanitize），完全沒測的是 admin CRUD，而金鑰處理的程式碼全在那裡。風險分布是反的。

## 1.4 `PUT /api/admin/settings` 邊驗證邊寫入 —— `settings.ts:157-272` 🟠

（初版第 1 項，**確認仍在**，位移。）14 個設定鍵逐一「驗證→立刻寫入」，任一失敗前面全部留在 D1，無 rollback。前端把 400 當整包失敗不重抓 → UI 與資料庫不一致且使用者不知道。

**修**：全部驗證 → 收集 statements → 一次 `env.DB.batch()`。副作用是十幾次 D1 round-trip 收成 1 次。
**測**：多鍵請求＋後段鍵不合法 → 斷言前段鍵**未**變動。（現有 `test/int/settings.test.ts:73` 只送單一鍵，從沒觸及這條路徑。）

## 1.5 上游錯誤內文可能含第三方金鑰 —— `observe.ts:25` 🟠

## 1.9 設計缺陷：不是現存漏洞，但要修 ＋ 寫進稽核

這兩條的共同點是**「目前打不到，但錯的方向是往開的那邊倒」**。稽核文件裡這類比真漏洞更能展示判斷力。

### `adminOk` 在 `LOGS_TOKEN` 未設時無憑證放行 —— `src/lib/auth.ts:246-248`

```ts
  if (env.LOGS_TOKEN) {
    if (await tokenEqual(token, env.LOGS_TOKEN)) return true;
  } else if (isLocal(url)) return true;      // ← LOGS_TOKEN 沒設 → 只看 Host
```

`isLocal`（`auth.ts:32`）判斷的是 `url.hostname`，在 Workers 裡來自 **Host 標頭——客戶端可控**。正式站有設 secret 且 Cloudflare 按 Host/SNI 路由，所以現在打不到。

但這個模式有兩個問題：**拿客戶端可控的標頭做授權判斷**，而且**在設定缺失時往「開」的方向倒**（fail-open on config error）。`pgUser`（`playground.ts:60`）的版本較軟，至少要求有 bearer token。

**修**：改成閘在 `wrangler dev` 才有的環境旗標，而不是 Host 標頭。整類問題一次消失。

### `/api/csp-report` 是無驗證的 D1 寫入口，且直通告警管道 —— `src/routes/api/csp-report.ts`

有 10% 取樣與長度上限（`:9-10, :22`），永遠回 204。但它是**匿名、無限流**的路徑，INSERT 進 `errlog`——正是 `tgAlertScan` 每 5 分鐘撈去推 Telegram 的那張表（`cron.ts:41-64`），而 `cron.ts:50` 把 `msg` 內插進訊息本體。

後果：持續 POST ＝ D1 寫入放大 ＋ **攻擊者可控文字直接進管理員的 Telegram**。

**修**：`RATE_LIMITER` DO 已經在了，掛上去即可（per-IP，比照 `demo.ts:112` 的用法）。

# 第 3 部分：稽核文件與敘事（約 3 小時）

## 3.1 `docs/AUDIT-2026-07.md` —— **重新定位，這是初版最大的改動**

初版想寫成：「我找到 3 個真缺陷，並把 4 個初判漏洞正確地推翻。」

**不要這樣寫。** 那等於公開宣告「我的 review 很嚴謹」，然後邀請面試官來 review 你的 review——而第二輪稽核在 20 分鐘內找到了 6 個漏網之魚。改成：

> **第一輪**稽核找到 3 個缺陷、推翻 4 個誤判（每個附推翻依據）。
> **第二輪換方法重跑，找到第一輪漏掉的 2 個真缺陷**，其中 `safeNext` 的開放轉址就在第一輪正在讀的 `auth.ts` 裡，只差 158 行。
> **我因此改了方法**：第一輪是「逐檔精讀」，會產生「讀過這個檔案就以為讀完了」的盲點。第二輪改成「逐**類別**掃」（所有進 `Location` 的值、所有會 throw 的解碼、所有 `SELECT *` 進回應的路徑），才抓到它。

**「我量了我自己稽核流程的漏檢率，並改了方法」比「我做了一次完美的稽核」強一個等級**，而且無法被證偽——因為你先自首了。這是資深工程師的動作，也正好補上這個專案「技術難度」之外的缺口。

**必須寫進去的內容：**

- 第一輪的 4 個推翻（`goodOrigin` localhost 有測試背書／前端未進工具鏈 ADR-0008:29 已揭露／前端「四個 bug」是型別推論抱怨不是缺陷／`cc.cd` 在 PSL 上）
- **PSL 那條要標成「查證救了自己第二次」**——它同時降級了 1.2 的 `getCookie` 從跨站攻擊到自傷
- 第二輪的方法論改動與新發現（1.1／1.2／1.3／1.8 的 `n`／1.9 兩條）
- **1.9 兩條要單獨開一節「不是漏洞，但方向錯了」**——「目前打不到」與「設計是對的」是兩件事。`adminOk` 在設定缺失時 fail-open、`csp-report` 讓匿名文字直達管理員告警管道，兩者都還沒被利用過，但**錯的方向是往開的那邊倒**。分辨得出這個差別，比多列一條真漏洞更能展示判斷力
- **測試套件自我檢查**：`test/int/router.test.ts:113-116` 的測試名稱是「管理員 API 帶金鑰 → 200」，斷言卻是 `expect([200, 401]).toContain(r.status)`——**授權成功或失敗都會通過**，而這是 `api/admin/users/index.ts` 在 vitest 裡唯一的接觸點。修它 20 分鐘。
- **E2E 缺口**：`e2e/site.spec.ts` 的 5 條流程涵蓋發文、跨身分串流、VPN 隱形、demo 配額、api-docs——**唯獨旗艦功能 `/relay` 完全沒有 E2E**。誠實列出，不一定要補
- 格式：發現／可利用性／`file:line` 證據／**判定（缺陷 or 取捨 or 誤判 or 方向錯）**／修補 commit 或推翻依據

- **`goodOrigin` 對正式網址放行 localhost Origin**（`auth.ts:237`，`test/unit/auth.test.ts:88-91` 記錄意圖）。風險真實但很窄：需要受害管理員載入攻擊者控制的 localhost 頁面。寫清楚什麼換什麼、什麼情況會改變決定，**不改程式碼**。
- **備份把明文機密寫進 R2**（`cron.ts:91-108`）：`BACKUP_TABLES` 對 `relay_channels`／`users`／`settings`／`vpn_channels` 都用 `cols:"*"`，所以每日 JSONL 含上游明文 API key、TG bot token、全體會員 VPN token，保留 14 份。`media.data` 有為 CPU 細心排除，機密沒被考慮過。**這不是 code bug**（金鑰本來就得明文存 D1 才能轉發），但 14 份滾動副本躺在另一個存取控制面不同的儲存系統，是實打實的爆炸半徑放大。
- **`vpn_token` 明文存**（`0001_baseline.sql:103`）：`sessions.sid` 與 `api_key_hash` 都雜湊，schema 註解還寫了理由，同一張表的 `vpn_token` 是明文而它同樣是 bearer credential。查詢是精確比對，改雜湊完全可行、索引照用。**配上上一條，一份 R2 備份 = 全體會員的 VPN 訂閱。**
- **修正 §2.4**：「path 片段重新編碼」防 traversal 的宣稱不精確——`encodeURIComponent` 不轉義 `.`（`relay/[[path]].ts:83-86`）。host 不可控、僅在已授權上游內。

# 第 4 部分：發版之後（這才是 3 個月的主戰場）

**做完上面就發 v2.1.0 tag ＋ Release，然後不要再碰它。**

這個專案有三個結構性天花板——技術難度、規模、協作訊號——**沒有一個能靠繼續做它來解決**。對新鮮人來說前兩個扣分有限，但第三個是免費可得的。

**下一步的優先順序：**

1. **OSS 貢獻（3 個被 merge 的 PR）** — LiteLLM／one-api／Cloudflare workers-sdk。你剛花 14 天把 LLM gateway、SSE 串流、配額計量摸熟，領域知識還熱著，而這些專案就活在同一個領域；`docs/COMPARISON.md` 證明你已經讀過他們的 codebase。**三個 merged PR 同時解掉規模與協作兩個天花板**——規模是別人的，協作訊號就是 review thread 本身，而且有 maintainer 幫你背書。3 個月足夠。
2. **把 `fastsse` 抽成獨立套件** — 57 行的核心 ＋ benchmark ＋ README。1–2 天的專案，產出的作品集價值高於在 uaip 上再花兩週。它現在是整個 repo 最有辨識度的東西，卻埋在一間雜貨店裡。
3. **硬體專案** — 先確認它跟你要投的職位是同一條線。**如果投後端／基礎設施，它是分散注意力；如果投嵌入式／韌體，那 uaip 跟職位幾乎無關、硬體專案才是主秀。** 兩個互不相關的中等專案，弱於一個強專案 ＋ 深度。這題先想清楚再開工。

# 明確不做，以及為什麼

這些是**好工程**，但只影響「這個專案夠不夠完善」，不影響「能不能幫你拿到工作」。留下判斷紀錄，不是留待辦。

| 不做 | 否決理由 |
|---|---|
| 抽 2,350 行前端 JS 出樣板字串 | `ADR-0001:31-32` 訂的閾值是「單頁 inline JS 超過 ~1,000 行」，最糟的 `PG_JS` 是 480 行，照專案自己的標準還沒到期。且 `ssr-smoke.test.ts` 的 `expectScriptsParse` 就是為它寫的護欄——**那是強項不是弱點** |
| 拆 `chat.ts:75` 的 481 行函式 | 它的註解密度證明作者完全理解它。拆它要 1 天、零可見收益、且是串流路徑（迴歸風險最高）。**但必須能在 30 秒內講出「為什麼不拆」** |
| 加資料存取層（160 個 `.prepare()` 散在 51 檔） | 「規模到了才痛」的債，你的規模沒到 |
| 修那 4 個前端「bug」 | 經逐行驗證程式碼是對的，只有型別推論會抱怨。改了零行為變化 |
| `tsconfig.client.json` ＋ `checkJs` 棘輪 | `ADR-0008:29` 已明文揭露前端不轉換，不是隱藏落差 |
| Istanbul 覆蓋率 ＋ 徽章 | 徽章不會讓任何人想面你。**但 1.3 的兩條 `maskRow` 測試要補**——那不是覆蓋率問題，是風險位置問題 |
| Router 層 admin 授權接縫 | 稽核逐一確認 33 個 handler 目前全都有檢查，沒有現存漏洞。防未來不是修現在 |
